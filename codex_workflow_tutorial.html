工作中正确使用 Codex 的教程

这份教程不讲空泛概念，只讲能落地的用法：什么时候该让 Codex 帮你， 什么时候必须人工把关，怎样提问才高效，怎样把输出稳稳接入团队流程。

一、先建立正确心智模型

先把 Codex 放在合适的位置，后面的效率和质量才有基础。

它擅长什么

梳理代码、生成样板、改写重复逻辑、补测试、解释报错、做跨文件影响分析。

代码理解 重复劳动 测试补齐

它不擅长什么

不清楚业务背景时的主观决策、涉及合规/安全/财务的最终判断、缺少上下文时的猜测。

需要人工确认 高风险决策

你要负责什么

定义问题，提供上下文，筛选方案，验证结果，决定是否合入。AI 不能替代你对结果负责。

最终责任在人

二、正确工作流：先约束，再生成，再验证

真正高效的用法，不是让 Codex 一次性“猜对”，而是把不确定性尽量前置消灭。

先定义任务 例如“补一个接口的单测”比“帮我看看这段代码”有效得多。
再补上下文 贴相关文件、报错栈、接口约定、已有测试、业务限制。
然后限制答案 指定“只改这 2 个文件”“先给方案，不直接改代码”“输出 diff 风格”。
最后做验证 检查逻辑、跑测试、看边界情况，避免把建议直接当结果。

三、一个高质量提问的结构

1

说明目标

先说你想得到什么结果，例如修 bug、补功能、优化性能、重构、写测试。

2

给出上下文

贴出相关代码、错误日志、接口返回、数据结构、调用链或文件列表。

3

列出约束

例如不能改数据库表、不能动公共接口、必须兼容旧版本、不能引入新依赖。

4

定义输出形式

你希望它给方案、直接修改、还是先列风险。把输出样式先说清楚，结果会稳很多。

示例：差的提问 vs 好的提问

差：
帮我改一下这个登录功能。

好：
我在 Vue 项目里修登录页的校验问题。
现象：输入空密码时没有提示，提交后接口也被调用了。
约束：不能改后端接口，只能改前端表单逻辑。
请先分析原因，再给最小修改方案，最后补一个测试点清单。
结论：Codex 不是读心器。你给的信息越像真实工单，它越像一个能干的同事。

四、常见工作场景怎么用

下面这些场景，基本覆盖日常开发里最值得让 Codex 上场的部分。

1. 接手陌生代码

让 Codex 先帮你画出模块边界、调用链和关键文件，迅速建立认知。

请先阅读这些文件：
- src/views/Login.vue
- src/services/auth.ts
- src/store/user.ts

输出：
1. 登录流程图
2. 每个文件的职责
3. 可能的风险点
4. 新人接手时最该先看的 3 个位置

2. 修 bug

把报错、复现步骤、预期结果和实际结果一次讲清，要求它先分析再改动。

复现步骤：
1. 打开订单页
2. 选择筛选条件
3. 点击导出

问题：导出按钮没有反应。
请先判断是事件没绑定、状态没更新，还是请求被拦截。
如果要改代码，只允许修改当前页面和相关 service。

3. 写测试

让它根据真实边界条件补单测，而不是只写“理想输入”。

请为这个函数补单元测试：
- 正常输入
- 空值
- 非法值
- 边界值
- 异常抛出

要求：使用现有测试框架，不引入新库，尽量覆盖分支。

4. 重构代码

先要它拆出重构目标，再给最小改动方案，最后再动代码，避免大刀阔斧地改坏逻辑。

请把这个组件拆成：
1. 数据处理层
2. UI 展示层
3. 可复用 hook / 工具函数

限制：
- 不改变页面行为
- 不新增依赖
- 先输出拆分建议，再输出改造步骤

5. 写文档

让它把复杂逻辑翻译成“给新人能看懂”的语言，尤其适合 README、操作手册和交接文档。

请根据这段实现写一份说明文档：
- 这个功能的目的
- 用户如何使用
- 常见失败原因
- 排查路径
- 修改时的注意事项

6. 代码审查前预检

在发起 review 之前，让 Codex 先帮你找潜在问题，减少无效往返。

请从以下维度检查这次改动：
- 逻辑正确性
- 边界条件
- 性能风险
- 安全风险
- 测试缺口

重点输出：可能被 reviewer 挑出来的问题。

五、直接可用的提示词模板

把下面的模板复制到日常工作里，再按自己的场景替换内容。

模板 A：先分析再动手

适合你自己还不确定问题在哪时使用。

请先不要直接修改代码。
基于我提供的文件和现象，先判断问题根因，列出 2-3 个可能方案，
对每个方案说明代价、风险和适用条件，最后推荐一个最小改动方案。

模板 B：限定改动范围

适合线上问题或多人协作时，避免 AI 改到不该碰的地方。

只允许修改以下文件：
- xxx.vue
- xxx.ts

不要改接口定义，不要新增第三方依赖。
如果你认为必须改其他地方，先解释原因并等我确认。

模板 C：输出交付物

适合把 Codex 当成一个能交作业的助手，而不是只给零散建议。

请输出以下内容：
1. 问题分析
2. 修改方案
3. 需要检查的测试点
4. 可能的副作用
5. 给团队成员的简短说明

六、最常见的坑

很多“Codex 不好用”的体验，本质上是使用方式不对。

应该这样做

  • 把任务拆小，一次只让它解决一个明确问题。
  • 提供真实上下文，不要只给片段化描述。
  • 让它先分析，再改代码，再总结风险。
  • 对结果做测试和人工审查。
  • 高风险场景保守使用，尤其是安全、财务、权限相关逻辑。

不要这样做

  • 把模糊需求直接丢给它，期待一次性命中。
  • 不看输出就直接复制到主分支。
  • 让它同时改太多文件，还不说明边界。
  • 忽略测试和回归验证，只看“代码像不像对的”。
  • 在无法验证的场景里，把它当成事实来源。

七、团队里更稳的协作方式

如果是多人协作，建议把 Codex 放进“辅助审查和加速执行”的位置，而不是“自动提交”的位置。

交付前

先让 Codex 帮你做预检，发现逻辑漏洞、遗漏测试和文档缺口。

评审中

把它当成第二双眼睛，帮助你提炼问题描述，缩短 reviewer 理解成本。

交付后

让它帮你沉淀成模板、FAQ、排障指南，把一次性经验变成团队资产。

八、最后记住这三句话

  • Codex 的价值在于把你从重复劳动里解放出来，不在于替你做判断。
  • 提问质量决定输出质量，约束越清楚，结果越可控。
  • 任何真正要进生产的改动，都必须经过人工复核和测试验证。
